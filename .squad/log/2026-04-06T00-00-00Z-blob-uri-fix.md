## Summary

Fixed a critical bug where `ProcessJob` in `JobWorker.cs` was throwing `Azure.RequestFailedException: The specified container does not exist` when running locally with Azurite. The root cause was three methods using manual URI parsing that worked for Azure production URLs but failed for Azurite's unique URI format.

### Key Achievement

Replaced all three manual `AbsolutePath.TrimStart('/').Split('/', 2)` URI parsers with `BlobUriBuilder` from the Azure Storage SDK, achieving correct URI parsing for both production Azure and local Azurite environments.

## Root Cause Analysis

**Problem:** Three methods in the codebase were parsing blob URIs using manual string manipulation:
```csharp
var parts = new Uri(blobUri).AbsolutePath.TrimStart('/').Split('/', 2);
var containerName = parts[0];
var blobName = parts.Length > 1 ? parts[1] : "";
```

**Why It Failed:** This approach assumes the first path segment after the hostname is the container name. This assumption holds for Azure production URLs:
- `https://account.blob.core.windows.net/container/blob` → container = "container" ✅

But it breaks for Azurite's emulator format:
- `http://127.0.0.1:10000/devstoreaccount1/container/blob` → container = "devstoreaccount1" ❌

The Azurite URI includes the storage account name as the first path segment, which was incorrectly parsed as the container name.

**Impact:** When running locally with Azurite, the code would attempt to access a container named "devstoreaccount1" instead of "container", resulting in 404 errors.

## Solution

Replaced all three parsers with `BlobUriBuilder` from `Azure.Storage.Blobs`:

```csharp
var blobUri = new BlobUriBuilder(new Uri(blobUri));
var containerName = blobUri.BlobContainerName;
var blobName = blobUri.BlobName;
```

`BlobUriBuilder` is the idiomatic SDK approach that correctly handles:
- Azure production URLs (`https://account.blob.core.windows.net/...`)
- Azurite emulator URLs (`http://127.0.0.1:10000/...`)
- Any other compliant blob storage URI format
